## 一、30 秒电梯演讲（一句话介绍）

> 我做了一个面向企业私有文档的知识库问答平台。它支持上传 PDF/Word/TXT 等文档，用「稠密向量 + BM25 关键词」混合检索加上 BGE 重排来精准召回，再用大模型生成**带原文溯源、防幻觉**的答案。在此之上，我用 LangGraph 实现了一个 ReAct Agent，能自动判断是否检索、拆解复杂问题、多轮迭代补充检索。系统还自研了 PII 脱敏、对话摘要、工具重试三个中间件，前后端用 FastAPI + Gradio，一键启动可部署。

---

## 二、项目背景：解决什么问题

直接把文档丢给大模型问答会遇到几个核心痛点，本项目逐一解决：

| 痛点 | 解决方案 |
|------|----------|
| 大模型不知道企业私有知识 | RAG：检索私有文档片段作为上下文 |
| 纯向量检索对关键词/专有名词召回差 | 混合检索：向量 + BM25 双路 |
| 两路结果质量参差 | RRF 融合 + CrossEncoder 重排 |
| 模糊问题检索不准 | HyDE 查询改写 |
| 大模型编造答案（幻觉） | 强约束 Prompt + 强制基于检索片段 + 溯源 |
| 简单问答无法处理复杂多跳问题 | LangGraph ReAct Agent，拆解 + 多轮检索 |
| 隐私泄露、长对话超限、接口抖动 | PII 脱敏 / 对话摘要 / 重试 三大中间件 |

## 三、整体架构

### 数据流：文档入库

```
上传文件 → 解析(PDF/Word/TXT) → 文本清洗 → 自适应语义分块
        → 内容哈希去重 → BGE 向量化(分批) → Chroma 入库(upsert 幂等)
```

### 数据流：普通 RAG 问答

```
用户问题 →(可选 HyDE 改写)→ 混合检索(向量+BM25) → RRF 融合
        → BGE Reranker 重排 → 阈值过滤 → 拼接上下文 → LLM 生成(防幻觉) → 带溯源回答
```

### 数据流：Agent 问答（LangGraph）

```
decide(是否检索) ─否→ direct(直接回答)
        │是
        ▼
   decompose(拆解子问题) → retrieve(多子问题检索) → reflect(信息是否充足)
        ▲                                              │不足且未达上限
        └──────────────────────────────────────────────┘
                                                       │充足
                                                       ▼
                                                  generate(生成答案)
```

### 分层目录

```
api/        FastAPI 接口层 + 可复用服务函数
agent/      LangGraph 状态、ReAct 图、LLM 工厂、HyDE
rag/        文档解析分块、向量库、混合检索+重排
middleware/ PII脱敏、对话摘要、工具重试
ui/         Gradio 双模块界面
utils/      配置、日志、通用工具
```

## 五、核心技术点深入（面试重点）

### 1. 混合检索（Hybrid Retrieval）

**为什么要混合？**
- 稠密向量检索：擅长语义相似，但对精确关键词、专有名词、数字编号召回差。
- BM25 稀疏检索：基于词频，擅长关键词精确匹配，但不懂语义。
- 两者**互补**，混合后召回更全。

**实现要点**：
- 两路各召回 `top_k * 3` 个候选，给融合和重排留空间。
- 中文 BM25 必须用 jieba 分词，不能按空格切分。
- BM25 索引按向量库片段总数变化判断是否需要重建（缓存优化）。

### 2. RRF 融合（Reciprocal Rank Fusion）

**公式**：对每个文档，`score = Σ 1 / (k + rank)`，其中 k 通常取 60，rank 是它在某一路结果中的排名。

**为什么用 RRF 而不是分数加权？**
- 向量相似度分数和 BM25 分数**量纲不同**，直接加权要归一化、调权重，麻烦且不稳。
- RRF **只看排名不看绝对分数**，无需归一化，对两路分数分布不敏感，实现简单、效果稳健。

### 3. 重排 Rerank（CrossEncoder）

**Bi-Encoder vs Cross-Encoder**：
- Embedding 是 Bi-Encoder：query 和 doc **分别**编码成向量算相似度，快但精度有限（可离线建库）。
- Reranker 是 Cross-Encoder：query 和 doc **拼在一起**进模型，输出相关性分数，精度高但慢（不能预计算）。

**两阶段策略**：先用快的 Bi-Encoder 粗召回一批，再用慢但准的 Cross-Encoder 对少量候选精排，兼顾速度与精度。

### 4. HyDE（Hypothetical Document Embeddings）

**思路**：模糊问题直接检索效果差，先让 LLM 针对问题**生成一段假想的答案文档**，再用「问题 + 假想文档」去检索。因为假想文档和真实答案文档在语义/用词上更接近，召回更准。

**代价**：多一次 LLM 调用，所以做成可开关。失败时回退为原问题。

### 5. LangGraph ReAct Agent

**普通 RAG vs Agent 的区别**：
- 普通 RAG：固定流程「检索→生成」，一问一检索。
- Agent：动态决策，能判断「要不要检索」「问题要不要拆」「信息够不够、要不要再检索」。

**状态图节点**：
- `decide`：判断是否需要检索（闲聊就不检索）
- `decompose`：复杂问题拆成 1~3 个子问题
- `retrieve`：对每个子问题检索并累积去重
- `reflect`：反思信息是否充足，不足则生成补充查询
- `generate` / `direct`：生成最终答案

**防死循环**：用 `max_iterations` 限制最大检索轮数。

### 6. 防幻觉与溯源

- **Prompt 强约束**：明确要求「只能用参考资料、资料不足要说无法回答、禁止编造」。
- **溯源**：每个检索片段带 `source`（来源文件）元数据，回答下方折叠展示「检索参考原文片段」，可核对。

### 7. 三大中间件

| 中间件 | 作用 | 关键实现 |
|--------|------|----------|
| **PII 脱敏** | 掩码邮箱/手机号/身份证/银行卡 | 正则匹配，输入输出**双向**脱敏 |
| **对话摘要** | 上下文超 token 阈值时压缩历史 | 估算 token，保留最近 N 轮 + 早期摘要 |
| **工具重试** | 接口抖动自动重试 | **指数退避 + 随机抖动**，可配最大次数 |

**指数退避公式**：`delay = min(base * factor^n, max_delay) * (1 ± jitter)`，抖动避免多个请求同时重试造成「重试风暴」。

### 8. 增量入库与去重

- 对每个 chunk 文本做归一化哈希作为 Chroma 的**文档 id**。
- 用 `upsert`：同 id 覆盖，天然实现**幂等**——同一文档重复上传不会膨胀向量库。

## 六、工程化与性能优化

| 优化 | 做法 | 效果 |
|------|------|------|
| 模型冷启动慢 | 启动时**后台线程预热**模型 | 首次问答不卡 |
| 弱网模型加载慢 | **本地缓存优先**(`local_files_only`) | 加载从几十秒→~3 秒 |
| 大文档入库无反馈 | 向量化**分批 + 进度回调** | UI 实时进度 |
| 模型重复加载 | 进程级**单例 + 加锁**(线程安全) | 只加载一次 |
| HF 下载难 | 启动前设 `HF_ENDPOINT` 镜像 | 免代理下载 |

## 七、面试题（按主题分类，含参考答案要点）

### A. RAG 基础

**Q1：什么是 RAG？为什么需要它？**
> 检索增强生成。先从知识库检索相关内容，再把内容作为上下文喂给 LLM 生成答案。解决大模型知识过时、不懂私有数据、易幻觉的问题，且无需重新训练模型。

**Q2：RAG 和微调（Fine-tuning）怎么选？**
> RAG 适合知识频繁更新、要溯源、私有文档场景，成本低、可解释；微调适合改变模型「风格/能力」、知识相对固定的场景。两者可结合。

**Q3：文档分块（chunking）为什么重要？怎么选 chunk 大小？**
> 太大：检索不精准、上下文浪费、易超 token；太小：语义被切碎、丢上下文。常用 300~800 字符 + 适当重叠(overlap)保留上下文连续性。本项目用 RecursiveCharacterTextSplitter，按段落→句子→标点递归切，尽量在语义边界切。

### B. 向量与检索

**Q4：Embedding 是什么？余弦相似度为什么常用？**
> 把文本映射成高维向量，语义相近的向量距离近。余弦相似度只看方向不看长度，适合文本语义比较；向量归一化后内积即余弦。

**Q5：向量库怎么做近似最近邻（ANN）？**
> 如 HNSW（分层可导航小世界图）、IVF 等。Chroma 默认用 HNSW，用图结构加速近邻搜索，牺牲少量精度换大幅提速。

**Q6：为什么要混合检索？只用向量不行吗？**
> 向量对精确关键词、专有名词、编号召回差，BM25 补这块；两者互补召回更全。（展开讲 RRF）

**Q7：RRF 的原理？为什么不直接加权分数？**
> 见上文第五章第 2 点：RRF 只看排名、无需归一化、对分数分布不敏感、稳健简单。

**Q8：BM25 是什么？**
> 基于词频(TF)和逆文档频率(IDF)的稀疏检索算法，对关键词精确匹配强，是 TF-IDF 的改进，引入文档长度归一化和饱和函数。

### C. 重排与优化

**Q9：Rerank 为什么能提升效果？Cross-Encoder 和 Bi-Encoder 区别？**
> 见第五章第 3 点。Cross-Encoder 让 query 和 doc 交互编码，精度高；Bi-Encoder 分别编码，快。两阶段：粗召回 + 精排。

**Q10：HyDE 原理？什么时候用？**
> 见第五章第 4 点。模糊/抽象问题用，代价是多一次 LLM 调用。

### D. Agent 与 LangGraph

**Q11：LangChain 和 LangGraph 区别？**
> LangChain 是组件库（LLM/检索/Prompt 等）+ 链式编排；LangGraph 是基于图的有状态编排框架，适合做有循环、有条件分支、需要记忆的 Agent，比线性 Chain 更灵活。

**Q12：什么是 ReAct？**
> Reasoning + Acting：让 LLM 交替「推理」和「行动（调工具/检索）」，根据观察结果决定下一步，适合多步任务。

**Q13：你的 Agent 怎么避免无限循环？**
> 用 `max_iterations` 限制最大检索轮数，reflect 节点判断信息充足或达上限就进入生成。

**Q14：Agent 模式比普通 RAG 好在哪？代价是什么？**
> 好：能处理复杂多跳问题、动态决策、按需多轮检索。代价：LLM 调用次数多、更慢、更贵，简单问题没必要。

### E. 工程化

**Q15：怎么防止大模型幻觉？**
> Prompt 强约束 + 强制基于检索内容 + 资料不足时拒答 + 溯源可核对 + 控制 temperature 调低。

**Q16：长对话上下文超限怎么处理？**
> 对话摘要中间件：超 token 阈值时把早期历史压缩成摘要，保留最近 N 轮。

**Q17：接口/工具调用失败怎么办？**
> 指数退避重试中间件，带随机抖动避免重试风暴，配最大重试次数。

**Q18：怎么保证增量入库不重复？**
> 内容哈希作文档 id + upsert 幂等。

**Q19：PII 脱敏怎么做的？为什么双向？**
> 正则匹配邮箱/手机号/身份证/银行卡并掩码。双向：输入脱敏防隐私进入模型/日志，输出脱敏防模型把检索到的敏感信息泄露出去。

### F. 系统设计 / 扩展性

**Q20：数据量大了（百万级文档）这套还能用吗？怎么扩展？**
> 单机 Chroma 有限。可换分布式向量库（Milvus/Qdrant）、Embedding 服务化、BM25 用 Elasticsearch、检索并行化、加缓存层、异步入库。

**Q21：怎么评估 RAG 系统效果？**
> 检索层：召回率、MRR、nDCG、命中率；生成层：忠实度(faithfulness)、答案相关性、有无幻觉；可用 RAGAS 等框架，也可人工标注。

**Q22：怎么做成多用户/生产级？**
> 加鉴权、按租户隔离向量库/命名空间、异步任务队列做入库、限流、监控日志、容器化部署。
